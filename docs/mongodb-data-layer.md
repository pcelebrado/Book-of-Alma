# SQLite Data Layer (Post-Migration)

## Status

This document replaces the previous MongoDB data-layer guidance.

- Decision: `DECISION_197`
- Migration date: `2026-03-06`
- Runtime datastore for web app: SQLite (`better-sqlite3`)
- Primary DB file: `/data/web.db`

## Why the migration happened

The template now aligns with OpenClaw's SQLite-native posture and removes MongoDB
as an application dependency.

- Lower runtime footprint and image complexity
- Fewer moving parts across Railway services
- Simpler persistence model for free-plan constraints

## Runtime topology

- `openclaw-web`: Next.js + SQLite (`/data/web.db`)
- `openclaw-core`: OpenClaw + QMD + SFTPGo

Critical deployment requirement:

- Web service must have a Railway volume mounted at `/data`.
- Without that volume, `web.db` is ephemeral and will be lost on redeploy.

## Schema and initialization

Schema is applied at startup from:

- `services/web/src/lib/db/schema.sql`

Connection and initialization logic:

- `services/web/src/lib/db/sqlite.ts`

Repository layer:

- `services/web/src/lib/db/repositories.ts`
- `services/web/src/lib/db/collections.ts` (compatibility exports)

## Data model

SQLite tables map 1:1 with prior application-level entities:

- `users`
- `book_sections`
- `book_toc`
- `notes`
- `highlights`
- `bookmarks`
- `reading_progress`
- `playbooks`
- `audit_log`
- `rate_limits`

Search support:

- FTS5 virtual table for section search with sync triggers.

## ID and compatibility contract

- IDs use 24-char hex format generated by `lower(hex(randomblob(12)))`.
- API payloads continue to expose `_id` for frontend compatibility.

## SQLite implementation patterns

- WAL mode enabled for read/write concurrency
- Foreign key enforcement enabled
- Busy timeout configured (5s)
- JSON fields stored as TEXT and parsed in repositories
- Upsert behavior implemented via SQLite-native patterns:
  - `ON CONFLICT ... DO UPDATE`
  - explicit SELECT + INSERT/UPDATE where needed

## Environment contract

Web variables:

- `SQLITE_DB_PATH=/data/web.db`
- `INTERNAL_CORE_BASE_URL=http://core.railway.internal:8080`
- `INTERNAL_SERVICE_TOKEN=<shared-token>`
- `AUTH_SECRET=<nextauth-secret>`
- `AUTH_URL=https://<web-domain>`
- `NEXT_PUBLIC_APP_URL=https://<web-domain>`

Removed from web contract:

- `MONGODB_URI`
- `BOOK_CANONICAL_COLLECTION`
- `BOOK_TOC_COLLECTION`

## Build and deploy notes

- Web Docker build includes native addon toolchain (`python3`, `make`, `g++`) for `better-sqlite3`.
- Next 14 standalone build must externalize native module:
  - `experimental.serverComponentsExternalPackages: ['better-sqlite3']`

## Verification checklist

1. `npm install` succeeds in `services/web`.
2. `npm run build` succeeds in `services/web`.
3. `GET /api/health` reports SQLite healthy.
4. CRUD checks succeed for notes/highlights/bookmarks/progress/playbooks.
5. Redeploy web service and confirm prior records still exist (volume durability).
